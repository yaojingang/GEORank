# 00. 重构原则

## 目标

GEOrank 2.0 的目标不是重写一个“看起来更新”的前端，而是完成一次长期可用的工程升级：

- 前端应用层从静态多页升级到可扩展架构
- 保留现有 FastAPI/Celery/数据库主链路
- 为多语言、统一鉴权、SDK、CLI 和后台演进提前铺路

## 核心原则

### 1. UI 保真优先

重构期间，新页面的首要目标不是“换设计”，而是：

- 保持现有页面的信息结构
- 保持现有页面的视觉气质
- 保持现有页面的交互路径

允许做的优化：

- 结构性整理
- 响应式稳定性提升
- 可维护性提升
- 多语言与 SSR 支持

默认不做：

- 无必要的大改版
- 功能流程重定义
- 视觉语言重置

### 2. 业务逻辑保真

现有五个前台业务模块必须在 2.0 中保持不变：

- 公司
- 诊断
- 方案
- 拓词
- 教程

后台业务模块保持一致：

- 公司管理
- 诊断管理
- 方案管理
- 拓词管理
- 教程管理
- 用户管理
- 系统设置

### 3. 后端能力保留

以下核心能力默认保留，不在 2.0 初期替换：

- FastAPI
- Celery
- Postgres
- Redis
- Qdrant
- Neo4j
- MinIO
- Playwright 爬虫链路

### 4. 并行新架构

2.0 不在原 `dist/` 上持续补丁式维护，而是在新目录下并行建设：

- `apps/web`
- `apps/admin`
- `packages/*`
- `cli/*`

旧系统在新系统未达到切换标准之前持续可用。

### 5. 契约先行

前端、后台、CLI 都必须围绕统一 API 契约建设。

唯一契约源：

- FastAPI OpenAPI

不允许在 2.0 里继续扩散：

- 页面内部零散手写 `fetch`
- 页面内部自行拼装错误结构
- 页面内部自行猜接口返回格式

### 6. 数据迁移先设计，后落库

多语言、slug、session、browser binding、翻译表等变更，必须先有文档和 Alembic 草案，不能边做边发明。

### 7. 仅在硬阻塞时暂停

默认自主推进，只有下列情况才暂停：

- 与用户未提交改动发生直接冲突
- 涉及不可逆数据迁移
- 需要变更已确认的业务规则
- 缺少外部密钥、账号或不可推断环境信息

## 切换定义

某个模块从 1.0 切换到 2.0，必须同时满足：

- UI 保真通过
- 业务链路通过
- API 契约通过
- 旧路径兼容通过
- 页面 SSR/SEO 输出通过
